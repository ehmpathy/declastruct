# vision: arg passthrough

> **root motto:** don't let declastruct interfere with --arg passthrough

## the outcome world

### before/after contrast

**before**: wish files are static. to target different environments, users resort to:
- multiple wish files (`wish.test.ts`, `wish.prod.ts`)
- environment variables set externally (`ENV=test npx declastruct plan ...`)
- hardcoded switches inside the wish file
- wrapper commands that configure state before invoking declastruct

**after**: wish files become parameterized. one file serves many contexts:

```bash
# same wish file, different targets — use -- to pass args to wish
npx declastruct plan --wish resources.ts --into plan.json -- --env test
npx declastruct plan --wish resources.ts --into plan.json -- --env prod
npx declastruct plan --wish resources.ts --into plan.json -- --env prod --region eu-west-1
```

### day-in-the-life

a developer manages aws resources across environments:

1. writes one `resources.ts` that parses `--env` from process.argv
2. runs `npx declastruct plan --wish resources.ts --into test.plan.json -- --env test` to preview test changes
3. runs `npx declastruct apply --plan test.plan.json` to apply
4. runs same commands with `-- --env prod` for production
5. ci/cd jobs pass `-- --env $DEPLOY_ENV` without custom wrappers

### aha moment

"wait, i can just use `--` to pass flags to my wish file? i don't need separate files or wrapper commands anymore."

the value clicks when the user realizes their wish file is now a reusable template, not a static declaration.

## user experience

### usecases

| goal | command | process.argv contains |
|------|---------|----------------------|
| target environment | `-- --env prod` | `['--env', 'prod']` |
| enable verbose mode | `-- --verbose` | `['--verbose']` |
| select region | `-- --region us-west-2` | `['--region', 'us-west-2']` |
| preview provider calls | `-- --preview` | `['--preview']` |
| filter resources | `-- --only user-*` | `['--only', 'user-*']` |

**note:** `--` separator is required. this ensures declastruct catches typos like `--wishe` instead of silent passthrough to the wish file.

### contract: process.argv injection

**how it works:**
1. commander parses known flags (`--wish`, `--into`)
2. args after `--` separator are captured as positional args
3. declastruct replaces process.argv with captured args
4. wish file is imported — top-level `parseArgs()` sees the injected args

**typo safety:** unknown flags (without `--`) trigger an error with a helpful hint:
```
error: unknown option '--env'
hint: to pass args to your wish file, use: -- --env
```

**the key insight:** wish files parse args themselves via standard node patterns. declastruct stays out of the way.

```typescript
// resources.ts — no declastruct-specific API needed
import { parseArgs } from 'util';

const { values } = parseArgs({
  args: process.argv.slice(2),
  options: {
    env: { type: 'string', default: 'test' },
    verbose: { type: 'boolean', default: false },
  },
});

export const getResources = async () => {
  const credentials = await loadCredentials({ env: values.env });
  return [
    new S3Bucket({ name: `app-${values.env}-assets`, ...credentials }),
  ];
};
```

**note:** a declastruct utility for common arg parse patterns will be available in the future.

### timelines

1. **immediate**: user adds `-- <args>` to command
2. **before import**: declastruct captures args after `--` and injects into process.argv
3. **on import**: wish file's top-level parseArgs() sees the injected args
4. **execution**: wish file uses parsed args to configure resources/providers

## evaluation

### pros

1. **typo safe**: declastruct catches typos like `--wishe` with helpful error
2. **familiar pattern**: `--` separator is standard (npm scripts, docker, etc.)
3. **zero magic**: args are just strings in process.argv, user parses as needed
4. **backwards compatible**: wish files that don't use args continue to work
5. **composable**: works with any arg parser (yargs, commander, parseArgs)
6. **single file**: eliminates need for per-environment wish files
7. **standard pattern**: wish files parse args like any node CLI tool

### cons

1. **parse burden**: user must parse args themselves (mitigated by future utility)
2. **extra typing**: must include `--` before wish args (but familiar from npm scripts)

### pit of success

| edgecase | behavior |
|----------|----------|
| no passthrough args | process.argv contains only node and entry paths |
| unknown flags without `--` | error with hint: "use: -- --flag" |
| `--` separator itself | stripped, not included in process.argv |
| args contain `--wish` after `--` | user's problem — they own their arg namespace |
| wish file ignores args | works fine — just don't call parseArgs |

**note on arg parse:** use a proper parser (parseArgs, yargs, commander) for robustness.

## open questions & assumptions

### assumptions

1. **[answered]** `--` separator is required — unknown flags trigger an error with a hint. this catches typos on declastruct flags.

2. **[answered]** process.argv injection is the approach — wish files parse args themselves via standard node patterns. declastruct stays out of the way.

3. **[answered]** both getResources and getProviders see the same args — yes, since args are in process.argv, both functions (and any top-level code) have access.

4. **[answered]** typescript consumers will want typed args — yes, but this is standard CLI behavior. strings are universal; users parse to types as needed.

### questions for wisher

1. **[answered] contract approach**: process.argv injection. wish files parse args themselves. a utility will be provided for common patterns.

2. **[answered] apply command**: apply should NOT support `--` args. reasoning: the plan file captures complete resource state at plan time. if apply used different args, it would be applying a different state than was planned — violates the plan/apply contract. if users need different credentials at apply time, they should use env vars (which are orthogonal to wish args).

3. **[answered] help text**: yes, declastruct should show `[-- <wish-args>]` in its help output for discoverability. example: `npx declastruct plan --wish <file> --into <file> [-- <wish-args>]`

### external research needed

- none — this is internal contract design

## what is awkward?

### arg parse is boilerplate

every wish file that uses args needs parse logic. this fights the "zero boilerplate" vibe.

**mitigation**: a declastruct utility for common arg parse patterns will be available in the future.

### plan/apply arg mismatch

user runs:
```bash
npx declastruct plan --wish r.ts --into p.json -- --env test
npx declastruct apply --plan p.json -- --env prod  # oops, different env
```

plan was created for test, but apply runs with prod args. this could be confusing.

**mitigation**: apply ignores `--` args entirely (uses plan's captured state). this is the decided behavior.
