# vision: declastruct --snap

## the outcome world

### before

```sh
# you run declastruct plan
npx declastruct plan --wish resources.ts --into plan.json

# plan.json shows decisions, but...
# - what did cloudflare actually return?
# - which fields were readonly? what were their values?
# - what was the raw state before the diff?

# you debug why a zone shows "no changes" but nameservers look wrong
# you have no visibility into what declastruct observed
# you add console.logs, re-run, squint at terminal output
# 20 minutes later you discover it's a readonly field
```

### after

```sh
# you run declastruct plan with --snap
npx declastruct plan --wish resources.ts --into plan.json --snap snapshot.json

# snapshot.json shows all that declastruct observed
cat snapshot.json | jq '.remote[0]'
{
  "forResource": {
    "class": "DeclaredCloudflareDomainZone",
    "slug": "DeclaredCloudflareDomainZone.example.com.abc123"
  },
  "state": {
    "_dobj": "DeclaredCloudflareDomainZone",
    "id": "abc123",
    "name": "example.com",
    "status": "active",
    "nameServers": ["gannon.ns.cloudflare.com", "tia.ns.cloudflare.com"],
    "originalRegistrar": "squarespace domains llc (id: 3827)",
    "createdOn": "2026-04-07T12:50:59.420278Z"
  }
}

# instantly see: nameServers value from cloudflare
# debug time: 30 seconds
```

### the aha moment

> "oh, i can just look at `snapshot.json` to see exactly what cloudflare returned — i don't need to add debug logs or re-run."

the value clicks when someone:
1. wonders why a field isn't updated
2. opens the snapshot
3. sees the full remote state (the field is readonly on the class)
4. understands immediately without code changes

## user experience

### usecases

| usecase | goal | how --snap helps |
|---------|------|------------------|
| debug | understand why field isn't diffed | see full remote state vs wished state |
| audit | prove what remote state was at plan time | `observedAt` timestamp + full state |
| drift detection | track changes over time | diff snapshots across runs |
| first-time setup | learn what the api returns | see real response shapes |
| incident response | "what did it see when it ran?" | snapshot is the receipt |

### contract

```sh
# snap full state into a file
npx declastruct plan --wish resources.ts --into plan.json --snap snapshot.json
```

```json

  "remote": [
```

### timeline

```
t0: user runs `declastruct plan --snap snapshot.json`
t1: declastruct fetches remote state (already happens today)
t2: declastruct serializes remote state into snapshot (NEW)
t3: declastruct computes diff (already happens today)
t4: declastruct writes plan.json (already happens today)
t5: declastruct writes snapshot.json (NEW)
t6: user inspects snapshot.json to debug/audit
```

## mental model

### how users describe it

> "it's like `terraform show` but for declastruct — you can see what the remote actually looks like"

> "it saves a receipt of what declastruct observed, so you can debug without re-runs"

> "it's the 'before' snapshot — what the world looked like when you planned"

### analogies

| analogy | maps to |
|---------|---------|
| terraform plan vs terraform state pull | plan.json vs snapshot.json |
| receipt | proof of what was observed |
| x-ray | see inside the black box |
| flight recorder | audit trail for incident response |

### terminology

| user says | we say |
|-----------|--------|
| "what did it see?" | remote state |
| "what did i ask for?" | wished state |
| "why didn't it change?" | readonly field |
| "when did it run?" | observedAt |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| debug | yes | full remote vs wished comparison |
| audit remote state | yes | observedAt + full state = complete receipt |
| drift detection | yes | diff snapshots over time |
| first-time setup | yes | see real api responses |
| no extra api calls | yes | declastruct already fetches this data |

### pros

- minimal performance cost (data already fetched; serialize + write is fast)
- opt-in (only writes if --snap flag present)
- machine-readable (json, easy to jq/parse)
- timestamp first (observedAt at top for quick identification)

### cons

- additional file to manage
- snapshot can be large for big infrastructures

### pit of success

| edgecase | how we handle it |
|----------|------------------|
| user forgets --snap | plan still works, just no snapshot |
| large infrastructure | snapshot is one file; use jq to filter specific resources |
| sensitive data in remote | user controls where snapshot goes, can gitignore |
| no remote state (new resource) | remote[].state is null, forResource shows what was looked up |

## open questions & assumptions

### assumptions (verified)

1. the remote state is available at the point where we'd serialize it (verified: `remoteState` in planChanges.ts)
2. wished state is the canonical version (after declastruct processes resources.ts)
3. `serialize()` from domain-objects stamps `_dobj` with class name

### decisions made

1. **--snap scope:** only resources in the wish
   - matches plan.json scope
   - remote-only resources aren't part of the plan

2. **terminology:** "remote" and "wished" in JSON
   - "remote" is familiar IaC term (terraform, pulumi)
   - "wished" connects to `--wish resources.ts`

3. **output format:** uses `serialize()` from domain-objects
   - stamps `_dobj` for class identification
   - matches plan.json structure (`forResource` + `state`)

4. **--snap always includes both:** remote[] and wished[]
   - no split flags needed
   - simple contract
