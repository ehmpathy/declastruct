# vision: after-plan apply hint

## the outcome world

### before

```bash
$ AWS_PROFILE=prep npx declastruct plan \
    --wish provision/aws.auth/account=prep/wish.ts \
    --into provision/aws.auth/account=prep/.temp/plan.json

🌊 declastruct plan
   ├─ wish: provision/aws.auth/account=prep/wish.ts
   ├─ plan: provision/aws.auth/account=prep/.temp/plan.json
   └─ snap: (none)

   ... planning output ...

🌊 declastruct plan
   ├─ resources: 4
   ├─ plan: provision/aws.auth/account=prep/.temp/plan.json
   └─ snap: (none)

$
```

user now thinks: "ok... so how do i apply? let me re-read the docs... what flags do i need?"

### after

```bash
$ AWS_PROFILE=prep npx declastruct plan \
    --wish provision/aws.auth/account=prep/wish.ts \
    --into provision/aws.auth/account=prep/.temp/plan.json

🌊 declastruct plan
   ├─ wish: provision/aws.auth/account=prep/wish.ts
   ├─ plan: provision/aws.auth/account=prep/.temp/plan.json
   └─ snap: (none)

   ... plan output ...

🌊 declastruct plan
   ├─ resources: 4
   ├─ plan: provision/aws.auth/account=prep/.temp/plan.json
   └─ snap: (none)

🥥 did you know?
   ├─ to apply, run
   └─ npx declastruct apply --plan provision/aws.auth/account=prep/.temp/plan.json

$
```

user adds their env vars (they know them) and runs. no syntax to remember.

### the aha moment

the value clicks when you realize:
- you don't need to remember apply syntax
- you don't need to swap --wish → --plan mentally
- the plan path is right there, ready to use


## user experience

### usecases

| goal | experience |
|------|------------|
| plan then apply | plan outputs ready-to-run apply command |
| review plan, apply later | copy the hint, come back later, paste |
| share with teammate | plan output is self-documenting |

### contract inputs & outputs

**what we transform:**
- `plan` → `apply`
- `--wish <file>` → removed (encoded in plan file)
- `--into <file>` → `--plan <file>`
- `--snap <file>` → removed (not needed for apply)
- passthrough args (`-- ...`) → removed (encoded in plan file)

**what we preserve:**
- invocation prefix (npx, pnpm dlx, yarn dlx, or bare)
- any other flags user supplied

**what we don't capture:**
- env vars (not in process.argv, user adds their own)

### timeline

1. user runs plan command with their env vars, flags, etc
2. plan executes, shows planning output
3. summary includes "to apply, run:" with exact command
4. user copy-pastes when ready to apply

## evaluation

### how well does it solve the goal?

✅ no need to remember apply syntax
✅ plan path is embedded in the hint
✅ simple, predictable output

### pros

- zero friction plan → apply
- preserves user's invocation context (prefix, flags)
- output documents itself

### cons

- requires argv parse to transform command
- user must add their own env vars (not detectable)

### edge cases → pit of success

| edge case | handling |
|-----------|----------|
| user ran via yarn/pnpm | preserve their prefix in transformed command |
| user had other flags | preserve them in transformed command |
| user had env vars | not detectable, user adds their own |
| passthrough args `-- foo` | removed (encoded in plan file) |


## open questions & assumptions

### assumptions (validated)

- env vars cannot be detected (process.argv doesn't include them)
- terraform pattern: show `terraform apply`, not env-prefixed command
- users remember their own env vars

### questions (triaged)

1. **env var access** [answered]
   - we don't capture them. user adds their own. (terraform pattern)

2. **multiline vs single line** [answered]
   - single line. cleaner, easier to copy. one triple-click to select.

3. **is simpler approach acceptable?** [answered by wisher]
   - wish says "only replace --wish and --into flags" to preserve "other flags"
   - **wisher confirmed:** must parse and transform, not static output
   - must preserve other flags user supplied (e.g., --verbose, custom flags)

4. **is npx the right default?** [answered]
   - npx works universally: finds global install or downloads
   - pnpm/yarn users can adapt if they prefer

5. **does format deviation matter?** [answered by wisher]
   - wish example shows `🌊 planned for 4 resources` but we keep current format
   - **wisher confirmed:** keep current format, just add the apply hint

6. **what if plan has no changes?** [answered]
   - still show the hint — user may want to run apply to verify
   - could say "0 resources" or skip hint — [wisher] could override

7. **should hint have blank line before it?** [answered]
   - no — keep it in the tree structure for visual consistency


## what is awkward?

### env var detection

env vars like `AWS_PROFILE=prep` are not in process.argv. we cannot preserve them in the transformed command. user must re-add them.

this matches terraform's behavior — acceptable tradeoff.

### argv reconstruction

we need to reconstruct the original command from process.argv. edge cases:
- quoted arguments with spaces
- flags with `=` vs space separators
- npx vs pnpm dlx vs yarn dlx detection

need to verify process.argv gives us enough info to reconstruct accurately.
